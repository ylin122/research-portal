# Code Review Baseline — research-portal

This file lists code-review findings the user has consciously decided NOT to fix (yet). The `code-review` agent reads this on every audit and moves matching findings to a "Known & Accepted" section at the bottom of its report instead of surfacing them at the top.

## Format

One entry per accepted finding. The `[F<id>]` token must match exactly the id the agent assigns in its report (snake-case from finding title).

```
- [F-<id>] file.jsx:line — short description (accepted YYYY-MM-DD, reason)
```

(none yet — populate after triaging the next code-review audit. Each entry should answer: which finding, when accepted, and why we're not fixing it now.)

## How to use

1. After a code-review audit, decide which findings you'll fix and which you're deliberately deferring.
2. For each deferred finding, copy its `[F-<id>]` and the file:line from the report into a new bullet here, with today's date and a one-sentence reason.
3. Future audits will move those findings to "Known & Accepted" automatically.
4. Remove an entry when you decide to fix it after all — the next audit will surface it again at full severity.
